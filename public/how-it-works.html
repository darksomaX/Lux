How Lux works

A short, plain explanation of what happens when you type a URL.

Lux is a web proxy. Your browser loads blocked sites by sending the requests through a server you control instead of contacting the site directly. The hard part is hiding that detour from the network censor and from the browser itself.

The four pieces

1. The service worker. When Lux opens, the page installs a small script called a service worker. That script sits between the page and the network. Every request the page makes passes through it first. The worker rewrites the destination into an obfuscated path and sends the rewritten request through the tunnel in step 3, instead of letting the browser fetch the site directly. The censor never sees the browser talk to the real site, so a block on that site never fires.
2. URL rewriting and obfuscation. A request for wikipedia.org turns into something like /service/hvtrs8%2F-wkkkpgdka.... The gibberish is the real URL run through an encoder. Lux defaults to an XOR encoder and lets you switch to a base64 "math" disguise, a plain form for debugging, or no path at all (iframe chains). Two things follow. The censor's URL filter does not see the word "wikipedia", and the address bar shows /service/... rather than the destination. The worker decodes the path on the way back so links inside the page keep working.
3. The wisp and Epoxy tunnel. The rewritten request cannot go straight to wikipedia.org, because that is what is blocked. Instead the service worker opens one WebSocket to /wisp/ on your server. The wisp protocol carries many streams inside that one socket. On top of wisp runs Epoxy (or libcurl, for the Scramjet engine), which performs the actual HTTPS and WebSocket exchange with the destination, including the TLS handshake. The net effect is that all proxied traffic looks like one ordinary encrypted wss:// connection to your own domain. To the censor it reads the same as any visit to a secure site.
4. The cloak. The Cloak button opens the proxied session inside a popup whose address reads about:blank and whose history is empty. Screen-share monitoring sees a blank tab. Tab disguise swaps the title and favicon to look like Google Classroom or a similar site. The panic key jumps to a decoy the moment you press it. Anti-close adds a leave warning.

What Lux gets past

  • IP and hostname blocklists (the most common kind): yes. The browser only ever talks to your server.
  • URL and SNI string filtering: yes. The destination hostname is encoded, and the only cleartext hostname is yours.
  • DPI that hunts for proxy traffic: harder. The tunnel is a normal WSS connection to a domain you control. Rotating the encoder and the domain makes fingerprinting harder still.
  • Logins (Gmail, OAuth): work, because all traffic is same-origin to your domain, so cross-site cookie and redirect flows resolve correctly. Epoxy and libcurl handle the TLS that a bare relay cannot.

What Lux does not get past

A censor that runs active probes (connecting to your server and testing whether it proxies) can still identify it. A censor that blocks your specific domain can stop you, which is why Lux is built to redeploy to a fresh domain in minutes. Lux is not anonymous like Tor. The server operator (you) can see connection metadata. For source protection, layer Lux with Tor or a VPN. Use it lawfully and ethically.

Run your own

git clone https://github.com/darksomaX/Lux
cd Lux
npm install
npm run build:uv
npm start        # http://localhost:8080

Deploy to Replit, Render, Railway, Fly.io, or any VPS. See the README for the configs. Vercel does not work for the full proxy: its serverless functions cannot hold the persistent WebSocket the wisp tunnel needs.
